volatile是jvm提供的轻量级的同步机制，具有三大特性：保证可见性、不保证原子性、禁止指令重排。

在了解三大特性之前，我们需要了解一下java的内存模型–JMM (Java Memory Model)，它是一种抽象的概念，并不真实存在。它描述的是一组规则或规范，定义的是程序里的各个变量的访问方式。

JMM中关于同步的三条规定：

  • 线程解锁前，必须把共享变量的值刷新回主内存
  • 线程加锁前，必须读取主内存的最新值到自己的工作内存
  • 加锁和解锁用的是同一把锁

图解：
图1
主物理内存中存在一个Student对象，它有一个属性age，初始值为15。存在t1和t2线程想要修改主内存中的age的值，但并无法直接修改主内存中的变量值，需要先将变量拷贝回自己的工作内存，先在自己的工作内存中进行操作。

图2
此时线程t1将自己工作内存中的变量值修改为12，接着就会将主内存中的值修改为12，同时此操作会被线程t2知晓，这也就是可见性的体现。

由于JMM操作的是实体是线程，而每个线程创建时JVM会为其创建一个工作内存（也就是栈空间）。工作内存是每个线程的私有数据区域，JMM规定所有变量都是储存在主内存中，即主内存是共享内存区域，所有线程都可以访问。

需要注意的是：线程对变量的操作都必须在自己的工作内存中进行，首先将变量从主内存拷贝到自己的工作内存，然后对变量进行操作，操作完成后再写回主内存。

因此，不同线程无法访问对方的工作内存，线程间的传值必须通过主内存来完成。

volatile的保证可见性： 

class Source{
    int number = 0;
    public void add(){
        this.number = 20;
    }
}

public class VolatileDemo {
    public static void main(String[] args){
        Source source = new Source();
        new Thread(()->{
            System.out.println(Thread.currentThread().getName()+"come in");
            try{
                TimeUnit.SECONDS.sleep(3);
            }catch (InterruptedException e){
                e.printStackTrace();
            }
            source.add();
            System.out.println(Thread.currentThread().getName()+" update value "+source.number);
        },"t1").start();

        while(source.number == 0){
            //main线程会一直循环读取number的值，直到number不为0
        }
        System.out.println(Thread.currentThread().getName()+" is over");
    }
}

结果为：
t1come in
t1 update value 20
三秒后t1线程修改了number的值为20，主线程一直都在循环中，程序无法自动退出。也就是，主线程一直不知道number已被改为其他值。

将变量number前加volatile: 

class Source{
    volatile int number = 0;
    public void add(){
        this.number = 20;
    }
}

public class VolatileDemo {
    public static void main(String[] args){
        Source source = new Source();
        new Thread(()->{
            System.out.println(Thread.currentThread().getName()+"come in");
            try{
                TimeUnit.SECONDS.sleep(3);
            }catch (InterruptedException e){
                e.printStackTrace();
            }
            source.add();
            System.out.println(Thread.currentThread().getName()+" update value "+source.number);
        },"t1").start();

        while(source.number == 0){
            //main线程会一直循环读取number的值，直到number不为0
        }
        System.out.println(Thread.currentThread().getName()+" is over");
    }
}

输出结果为：
t1come in
t1 update value 20
main is over
main线程不再一直在循环中等待，也就证明了volatile的保证可见性的特性。

volatile的不保证原子性：
原子性代表的是不可分割，保证完整性，要么做完，要么不做。 

class Source{
    volatile int number = 0;
    public void addPlus(){
        number++;
    }
}

public class VolatileDemo {
    public static void main(String[] args){
        Source source = new Source();
        for(int i = 0;i < 10;i++){
            new Thread(()->{
                for(int j = 0;j < 100;j++){
                    source.addPlus();
                }
            },String.valueOf(i)).start();
        }
        //等待上面的线程执行完成后，调用main线程查看最终的number的值
        //判断大于2的原因是：除了main线程外还有一个后台GC线程
        while(Thread.activeCount() > 2){
            Thread.yield();
        }
        System.out.println(Thread.currentThread().getName()+"current number is "+source.number);
    }
}

10个线程，每个线程执行100次number++的操作，如果volatile保证原子性的话，那最终的结果应该是1000。但是最终的运行结果不是，因此证明了volatile无法保证原子性。
那么为什么volatile不保证原子性呢？以下举例来解释：
首先有两个线程，二者均将主内存中的变量拷贝回自己的工作内存；
接着，当线程1调用addPlus()方法将自己的number值加1后(number=1)，正准备写回主内存，可此时因底层调度机制，线程1被挂起；
然后，线程2在自己的时间片中完成了addPlus()方法，且将结果写回了主内存，但来不及通知线程1，number的值已经被修改为1；
最后，线程1被激活则会继续做未完成的步骤，即将在自己工作内存中改变的变量值写回到主内存，这样就造成数据复写和数据丢失的后果了。

那么如何保证原子性呢？
1、使用sychronized对变量进行加锁
在多线程的环境下不推荐此种用法，因为其虽然能解决问题，但是并发性差。
2、使用AtomicInteger(原子性整形封装，实在java.util.concurrent.atomic包下的) 

class Source{
    AtomicInteger atomicInteger = new AtomicInteger();
    public void addAtomic(){
        atomicInteger.getAndIncrement();
    }    
}

public class VolatileDemo {
    public static void main(String[] args){
        Source source = new Source();
        for(int i = 0;i < 10;i++){
            new Thread(()->{
                for(int j = 0;j < 100;j++){
                    //source.addPlus();
                    source.addAtomic();
                }
            },String.valueOf(i)).start();
        }
        while(Thread.activeCount() > 2){
            Thread.yield();
        }
        System.out.println(Thread.currentThread().getName()+"current number is "+source.atomicInteger);//使用的是原子性的number值
    }
}

此时输出的结果为1000，即保证了原子性。

volatile的禁止重排序
先了解一下有序性的相关概念
计算机在执行程序的时候，为了提高性能，编译器和处理器常常会对指令做重排序的操作，一般分为以下几种：

  • 单线程环境里面确保程序最终执行结果和代码顺序执行的结果一致（即指令重排对单线程无影响）
  • 处理器在进行重排序时必须要考虑指令之间的数据依赖性（即逻辑顺序）
  • 多线程环境中线程交替执行，因编译器优化重排序的存在，多个线程使用的变量是否能保证一致性时无法确认的（结果无法预测）

指令重排：源代码按顺序排列，但是底层为了执行代码的效率能够得到提高，不一定也会按照源代码的编写顺序执行，尤其是多线程环境

例1： 

public void sort(){
  int x = 1;     //1
  int y = 2;     //2
  x = x + 5;     //3
  y = x + x;     //4
}

那此时语句4是否可以为第一句？那肯定是不行的，因为变量必须得先定义，这样会违背数据的依赖性。

例2: 

public class Demo{
  int a = 0;
  boolean flag = false;
  public void method(){
     a = 1;        //1
     flag = true;  //2
  }
  public void method01(){
      if(flag){
        a = a + 5;
        system.out.println("a = "+a);
      }
  }
}

上述代码在多线程情况下则有可能会出现：一个线程的语句2先被执行，另一个线程先执行了method01()方法，导致最后的a = 5，与正常顺序执行的a的结果不同，这样就没有抱枕该数据的一致性了。故需要用volatile来禁止指令重排。

以高并发下的DCL模式为例讲解禁止指令重排： 

public class SingletonDemo {
    private static SingletonDemo instance = null;
    private SingletonDemo(){
        System.out.println("无参构造方法");
    }
    public static SingletonDemo getInstance(){
        if(instance == null){
             instance = new SingletonDemo();
        }
        return instance;
    }

    public static void main(String[] args){
        for(int i = 0;i < 10;i++){
            new Thread(()->{
                SingletonDemo.getInstance();
            },String.valueOf(i)).start();
        }
    }
}

在单线程模式下调用getInstance()方法只会输出一个 “无参构造方法”。但是若是在多线程环境下测试，则会多次输出，这样就算不上是单例的概念了。那么接下来就介绍DCL模式，即双端检索机制的代码。 

public class SingletonDemo {
    private static SingletonDemo instance = null;
    private SingletonDemo(){
        System.out.println("无参构造方法");
    }
    public static SingletonDemo getInstance(){
        if(instance == null){
             synchronized (SingletonDemo.class){
                 if(instance == null){
                     instance = new SingletonDemo();
                 }
             }
        }
        return instance;
    }

    public static void main(String[] args){
        for(int i = 0;i < 10;i++){
            new Thread(()->{
                SingletonDemo.getInstance();
            },String.valueOf(i)).start();
        }
    }
}

instance = new SingletonDemo();可以分成三步完成创建对象操作
1、memory = allocate(); //分配对象的内存空间
2、instance(memory); //初始化对象
3、instance = memory; //设置instance第一步分配的内存空间
当执行的顺序变成1->3->2时，可知还未完成初始化怎能为初始化的对象指向地址。所以这种情况下还不是完全安全的，需要排除指令重排带来的问题，故可以使用volatile禁止指令重排的特性在instance前加volatile进行修饰，即： 

private volatile static SingletonDemo instance = null;

禁止指令重排的知识小结：
1、volatile实现禁止指令重排优化，从而避免多线程环境下程序出现乱序执行现象；
2、内存屏障(Memory Barrier)，又可以称为内存栅栏，它是一个CPU指令，起到保证特定操作的执行顺序和保证某些变量的内存可见性的作用；
3、由于编译器和处理器都可以执行指令重排的优化指令，若在指令间加一条内存屏障，则可以高斯编译器和处理器，不论什么指令都不能和这条内存屏障指令进行重排序操作，也就是说可以通过内存屏障禁止在其前后的指令进行重排序；
4、写内存屏障（Store Memory Barrier）：处理器将存储缓存值写回主存（阻塞方式）
读内存屏障（Load Memory Barrier）：处理器，处理失效队列（阻塞方式）
5、内存屏障种类
图3
6、volatile读前插读屏障，写后加写屏障，避免CPU重排导致的问题，实现多线程之间数据的可见性
7、对于处理器来说，内存屏障会导致cpu缓存的刷新，刷新时，会遵循缓存一致性协议:

  • 解锁时(lock)，jvm会强制刷新cpu缓存，导致当前线程更改，对其他线程可见;
  • 标记volatile的字段，在写操作时，会强制刷新cpu缓存，标记volatile的字段，每次读取都是直接读内存;
  • 编译器在final写操作后，会插入内存屏障，来禁止重排序，保证可见性。
